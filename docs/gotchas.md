# Non-Obvious Patterns & Gotchas

> Read the relevant entries BEFORE editing code in that area. Referenced from `CLAUDE.md`.
> Area index: payments → 1,2,6,7,12,13,14,15,21,22,34 · service lines / multiple plans → 1,16,25,41 · currency → 18,19,21,22,24,36,38 · branches → 26,27,28,29,30,31,32 · tiers → (see docs/features.md) · sales/products → 35,36,37 · auth/signup → 3,4,33,40 · i18n/UI → 5,11,17,39,44 · cross-slice → 9.

1. **Payments use upsert** — the `UNIQUE(customer_plan_id, billing_month)` constraint (per **service line**, not per customer) means re-paying a voided month upserts the existing row (nullifies `voided_at`), not inserts a new one. A customer with several lines can hold one payment per line per month; the upsert `onConflict` target is `customer_plan_id,billing_month`.

2. **Payment amount is a snapshot** — stored at recording time, never recomputed from `plan.price`. Historical amounts survive plan changes or deletions.

3. **`superadmin` role exists in DB but is not exposed in SubsTrack** — it appears in `UserRole` type but is filtered from the user management UI. Only the SuperAdmin app uses it.

4. **Login email is synthetic** — `username@tenantcode.com` is a convention for unique identification in Supabase Auth, not a real email.

5. **Language change reloads the app** — Arabic requires RTL layout via `I18nManager`, which needs a reload. This is handled in `storage.ts`.

6. **In-flight guard for double-tap** — stores check `if (get().loadingCreate) return` before proceeding to prevent duplicate payment submissions.

7. **Year-scoped payment loading** — `paymentStore` fetches payments only for the displayed year. Navigating years triggers new fetches (cached by year).

8. **`before_start` is a 4th month status** — months before `customer.startDate` are non-tappable and shown gray regardless of other logic.

9. **No cross-slice imports at module scope** — slice files may import types from peer slices but never the slice creator or per-slice hook. Cross-slice reads happen inside actions via `get().<otherSlice>`. Caller-supplied data (tier, usage, currency) still flows in as parameters from the component.

10. **SuperAdmin uses service role key** — `supabaseAdmin.ts` bypasses RLS entirely, giving full DB access. This is intentional for the SaaS owner's admin operations.

11. **Cairo font for Arabic** — loaded via `expo-font` at root layout; `useAppFont` hook applies it. The custom `Text` component selects font family per current language.

12. **`billing_month` is always YYYY-MM-01** — the first day of the month, always. Validated in PaymentService before insert.

13. **Multi-month payments store the bundle price, not per-month** — `payment.amount` is the total bundle price. The `durationMonths` field on the payment record tells the grid how many consecutive months are covered from `billingMonth` forward.

14. **`isGroupSecondary` flag in MonthEntry** — months 2+ in a multi-month payment block have `isGroupSecondary = true`. The grid uses this for visual merging (no gap between cells, "Included" sublabel) and to prevent double-tapping secondary months.

15. **Multi-month conflict resolution** — `createMultiMonthPayment` shifts `effectiveStart` and reduces `effectiveDuration` when leading months are already paid. The recorded payment starts at the first uncovered month, not the requested start, and covers only the remaining months in the original block.

16. **Non-regular customers never appear in unpaid counts** — dashboard `unpaidThisMonth` and the "Unpaid" tab filter only query regular (`is_regular = true`) active customers. Non-regular customers can still have payments recorded normally. With **multiple plans per customer**, these counts are aggregated across a customer's **active service lines**: `unpaidThisMonth` counts a customer once if any active line is uncovered this month (`CustomerRepository.countUnpaidForMonth` joins `customer_plans`); the list's fully-paid/partial badges (`findPaymentStatusForMonth`) require every active line settled for green.

17. **EmptyState first-data button** — `EmptyState` accepts an optional `onAction` + `actionLabel` prop. Lists (plans, customers, users) pass these to render a "Create First X" button when the list is empty and the user is not actively searching.

18. **`null currency_id` means USD** — every money column (`plans.price`, `payments.amount_due`/`amount_paid`, `users.display_currency_id`) treats `null` as USD. USD is never inserted as a `currencies` row.

19. **CurrencyInput does NOT convert on currency change** — switching the dropdown from USD to LBP keeps the typed `100` literal, reinterpreting it as `100 LBP`. This is correct because the user is saying "I meant this number in the other unit," not "convert what I typed."

20. **Display currency lives in AsyncStorage, not the DB** — `uiPrefStore.displayCurrencyId` (persisted via Zustand `persist` + `AsyncStorage`). There is no `display_currency_id` column on `users`. This keeps it a pure UI preference that doesn't round-trip through Supabase on every session restore.

21. **`rate_per_usd_snapshot` freezes payment USD value** — every payment row carries `rate_per_usd_snapshot`, the live `currencies.rate_per_usd` at the moment the payment was recorded. PaymentDetailSheet (receipt), CustomerPaymentPanel year total, and Dashboard aggregates all convert via this frozen rate, so editing a currency's live rate never retroactively shifts historical USD values. USD payments (`currencyId === null`) store snapshot = 1. **Plan prices** still use the live rate (forward-looking pricing — that's the desired behavior). The snapshot is captured at the boundary: `PaymentFormSheet` resolves the `Currency` from `useCurrencyStore` and passes it into `paymentStore.createPayment` / `createMultiMonthPayment`, which extract `currency?.ratePerUsd ?? 1` and forward to `PaymentService`. **Editing** a payment also re-snapshots: `PaymentService.updatePayment(id, amountDue, amountPaid, currency)` recomputes `ratePerUsdSnapshot` from the (possibly newly chosen) currency at edit time, so historical USD totals reflect the corrected record. Use the `paymentSnapshotCurrency(payment, currencies)` helper in `src/core/utils/currency.ts` when displaying a payment — it clones the live `Currency` with `ratePerUsd` overridden by the snapshot.

22. **Dashboard aggregates in USD using snapshots** — `DashboardService.getMetrics()` fetches `{amount, rate_per_usd_snapshot}` rows for the month and divides each by its snapshot before summing. The screen then re-formats the USD total into the user's display currency. No live-currencies lookup is needed for the sum — the snapshot is the rate.

23. **`authStore.restoreSession` / `login` prime the currency store** — after auth succeeds, `useCurrencyStore.fetchCurrencies()` is called so all downstream `CurrencyInput`s and formatters have data immediately. `logout` resets the currency store.

24. **`currencies.code` is uppercase A-Z, 2-8 chars, and never 'USD'** — enforced by a CHECK constraint and validated again in `CurrencyService.validate()`.

25. **Quick Pay `?quickPay=1` handshake** — the customer list opens `customers/[id]?quickPay=1` when a customer has no eligible fixed-price line to auto-pay (custom-price / plan-less). `CustomerPaymentPanel` reads the param, waits for the selected line's grid to load, then auto-selects the current-month entry and opens `PaymentFormSheet`. A `useRef` guard fires it once per mount; `router.setParams({ quickPay: undefined })` clears it. **With multiple plans**, the list quick pay (single or bulk) collects **all eligible fixed-price lines** at once via `paymentStore.bulkPayCustomers` (one `BulkPayCustomerRequest` per line, each carrying its `customerPlanId`) — it no longer calls `createPayment` / `createMultiMonthPayment` directly. The per-cell 3-dot menu (Open / Pay / Void) inside the panel still pays the tapped month of the **selected line** via `createPayment` / `createMultiMonthPayment`.

26. **`branch_id NULL` semantics** — on `users` it remains a real selectable state meaning "tenant-wide admin" (form picker is `nullable` only when `role === 'admin'`). On `plans` and `products`, NULL means **SHARED** — a catalog item visible to every branch; tenant-wide admins choose it freely (the `PlanFormSheet` / `ProductFormSheet` branch picker is `nullable={user?.branchId === null}`, and neither `PlanService` nor `ProductService` requires a branch). On `customers`, NULL is a **legacy-only** UNASSIGNED state: the form picker is `nullable={false}` and `CustomerService` rejects `branch_id = null` when the tenant has any active branch — the only way a customer ends up null today is if its branch was deleted (FK `ON DELETE SET NULL`). The header `BranchSelector`'s "Unassigned" filter exists to surface those orphaned customer rows for re-assignment.

27. **Tenant-wide admin = `users.branch_id IS NULL`** — there is no separate role for them. Any admin without a branch is effectively a "super admin" for the tenant. Multiple are allowed. The existing SaaS-level `superadmin` role (separate SuperAdmin app) is unrelated.

28. **Branch filter is purely UI state** — `uiPrefStore.currentBranchId` (persisted to AsyncStorage). RLS lets tenant-wide admins see everything; the app passes `?branch=X` filters per-query to narrow the view. Branch-scoped users ignore `currentBranchId` and always see only their branch (their `users.branch_id` is the only possible filter, enforced by RLS).

29. **`BranchSelector` is a compact chip embedded inside `PageHeader`** — it renders as a small pill (`rounded-full px-3 py-1`, gray when "All Branches" is active, indigo-tinted when filtered) directly under the page subtitle. `PageHeader` always renders `<BranchSelector />` internally — screens never instantiate it themselves. Dashboard has no `PageHeader`, so the chip is placed inside the greeting block (under the date line) in [DashboardScreen.tsx](../SubsTrack/src/modules/dashboard/screens/DashboardScreen.tsx). The chip's tap opens the shared `DropdownModal` exported from [Dropdown.tsx](../SubsTrack/src/shared/components/Dropdown.tsx) — same search/option-list UX as form-field dropdowns. Self-concealment is unchanged: returns `null` for branch-scoped users and for tenants with fewer than 2 active branches. The same "≥2 active branches" gate is exposed as `useIsMultiBranchActive()` in [BranchPicker.tsx](../SubsTrack/src/shared/components/BranchPicker.tsx) and shared with `BranchPicker` so every dropdown / picker / filter hides in lockstep.

30. **New tenants auto-get a "Default Branch"** — [TenantService.createTenant](../SuperAdmin/src/modules/tenants/services/TenantService.ts) inserts a `Default Branch` row right after the tenant row, before the admin auth user is created. `branches.tenant_id` has `ON DELETE CASCADE` so the existing rollback paths still clean up correctly.

31. **Single-branch tenants behave as if branches don't exist** — with exactly 1 active branch, `BranchSelector` + every `BranchPicker` hide. Form sheets auto-bind new records to that lone branch (`CustomerFormSheet`, `PlanFormSheet`, `UserFormSheet` for `role='user'`). `UserFormSheet` toggles `branchId` back to `null` when the role flips to `admin` because admins remain tenant-wide. The auto-fill applies only on _create_, never on edit. The Branches admin CRUD screen stays reachable so admins can add a 2nd branch and activate multi-branch UI.

32. **Branch is mandatory for customers and staff users once any branch exists** — `BranchPicker` is passed `nullable={false}` in `CustomerFormSheet` and (when `role === 'user'`) `UserFormSheet`. Submit buttons are disabled until a branch is picked, and the services re-validate (`CustomerService.validateInput`, `UserService.validateBranchAssignment`) using a `tenantHasBranches` flag that each store reads from `useBranchStore`. **Plans and products are NOT mandatory** — like products, plans can be created SHARED (null branch) by tenant-wide admins, so `PlanFormSheet` / `ProductFormSheet` pass `nullable={user?.branchId === null}` and neither service requires a branch. Tenant-wide admins (`users.role === 'admin'` with `branch_id = null`) are likewise exempt for customers/users. The 0-branch case (legacy tenants with no branches yet) skips enforcement entirely.

33. **Self-service tenant signup goes through an Edge Function, never direct inserts** — the SubsTrack mobile app ships only the anon key, and there is no INSERT policy on `tenants`/`branches`/`tier_plans`. The public [create-tenant](../SubsTrack/supabase/functions/create-tenant/index.ts) edge function (deployed with `--no-verify-jwt`) is the **sole** anon-accessible path for creating a tenant. It uses the service-role key to perform the full sequence with cascading rollback: lookup Free tier id → `tenants(tier_id=Free)` → `branches` ('Default Branch') → `auth.users` → `public.users` (role=`superadmin`, branch_id=null). The pre-check on the workspace screen uses `is_tenant_code_available` — a SECURITY DEFINER RPC granted to `anon` that returns a boolean (no row data, just a yes/no oracle). Self-signup mirrors the role assignment SuperAdmin's `TenantService.createTenant` uses (`superadmin`), which means the new owner is filtered from their own Users screen per note #3 — same behavior as tenants created from SuperAdmin today. Future paid-plan gating plugs into this edge function (the `paymentToken` field is already accepted-but-ignored in the request body) and into `subscriptionStore.upgrade()`.

34. **`partial` is a first-class `MonthStatus`** — when a payment covers a month and its `balance > 0` (amount_paid < amount_due), `PaymentService.buildMonthGrid` emits `status: "partial"` instead of `"paid"`. Amber cells in the grid replace the old paid-with-orange-dot rendering. Customer-list badges reflect the same: `paymentSlice` exposes **two** sets — `currentMonthFullyPaidIds` and `currentMonthPartialIds` — populated by `fetchCurrentMonthPaymentStatus()` (repository selects `balance` alongside `amount_paid`). Slice mutations (`createPayment`, `createMultiMonthPayment`, `updatePayment`, `voidPayment`) recompute set membership via the `applyPaymentStatus` / `clearPaymentStatus` helpers so editing a payment between full↔partial keeps the list badge correct. The unpaid tab + Quick Pay gating both treat membership in **either** set as "already has a payment record for this month." A `partial` month behaves like `paid` everywhere a payment record matters (`PaymentDetailSheet` opens on tap, `MonthGrid` multi-month grouping merges them, `PaymentFormSheet` treats them as multi-month conflicts, `CustomerPaymentPanel`'s `paidCount` includes them). The unpaid banner / unpaid tab only fire for `"unpaid"`. **Overdue override:** the customer-list badge shows `"unpaid"` whenever the customer has *any* unpaid past month — even if the current month is fully paid. `paymentSlice` exposes a third set, `overdueCustomerIds`, computed by `fetchOverdueStatus(customers, graceDays)` → `PaymentService.findOverdueCustomerIds`, which runs `buildMonthGrid` per active+regular customer across every year from `startDate` to now and flags any `"unpaid"` month (so the decision stays in `buildMonthGrid`, rule #1). `CustomerListScreen` refreshes it via `useFocusEffect` (keyed on `customers` + `graceDays`) so paying a past month in the detail panel updates the badge on return. The badge precedence is `overdue → unpaid`, then current-month `fully-paid → paid`, `partial → partial`, else `unpaid`; the unpaid tab includes a customer when `overdueCustomerIds.has(id) || !hasCurrentMonthPayment(id)`.

35. **Sales are independent from subscription payments** — `sales` and `payments` are separate tables with separate services, slices, and screens. They share zero code beyond the snapshot rate principle (`rate_per_usd_snapshot`). Don't try to unify them — subscriptions are month-shaped (`billing_month`, `duration_months`, multi-month coverage), one-off sales are not. The only place they intersect is `DashboardService.getMetrics()`, which sums both in USD using each row's frozen snapshot rate and returns `subscriptionRevenue` + `salesRevenue` + their combined `monthlyRevenue`.

36. **`paymentSnapshotCurrency()` works on any row with `currencyId` + `ratePerUsdSnapshot`** — despite the name, the helper in `src/core/utils/currency.ts` is shape-based, not payment-specific. Reuse it for sales (`SaleCard`, `SaleDetailSheet`) so historical USD-equivalent values stay drift-free when a tenant edits the live `currencies.rate_per_usd`. If you later add another snapshot-bearing entity (refunds, deposits), reuse it again rather than copying the pattern.

37. **`AsyncEntityPicker` replaces `Dropdown` only when the option list is too large to fit in memory** — `Dropdown` does client-side filtering against a pre-loaded options array (perfect for branches, currencies, plans). `AsyncEntityPicker` accepts a `loadPage(search, page)` callback, debounces input via `useDebounce` (300 ms), paginates via `FlatList.onEndReached`, and discards stale responses via a `requestToken` ref (same pattern as `customerSlice.searchToken`). Reach for it for customers (used in `SaleFormSheet`) and any future large-entity picker. Don't migrate small static-list dropdowns — `Dropdown` is fine there.

38. **`app_options` is global config + seeds the default LBP currency** — `app_options` is a single app-wide key/value table (no `tenant_id`), managed by the SaaS owner from SuperAdmin's **Options** tab (full CRUD, service-role writes) and read-only in SubsTrack (`options` module + `optionSlice`, fetched at app bootstrap **and** re-primed in `primePostAuth`, **not** reset on `logout`; RLS = `anon` + `authenticated` `SELECT`, since some flags gate pre-auth UI). The `LiraRate` row holds the default USD→LBP rate. Other seeded flags: `AllowPlanUpgrade` (swaps the upgrade button for a WhatsApp "contact to upgrade" deep-link to `SupportWhatsAppNumber`), `AllowSelfServiceSignup` (hides the login "Create workspace" button **and** is enforced in the `create-tenant` edge function). Read flags via the typed hooks in `useOptionSlice.ts` (`useCanUpgradePlan` / `useSelfServiceSignupEnabled` / `useSupportWhatsAppNumber`), never by poking `items` directly. On tenant creation BOTH paths — SuperAdmin `TenantService.createTenant` and the public `create-tenant` edge function — auto-seed an `LBP` currency (`decimals 0`, symbol `ل.ل`) whose `rate_per_usd` is copied from `LiraRate` (falling back to `DEFAULT_LIRA_RATE = 89000` when missing/invalid, so a bad option never blocks signup). The seed is a one-time starting value, not a live link — the tenant edits its own LBP rate afterward. Option keys are immutable after creation; reference them via `OPTION_KEYS`.

39. **Keyboard avoidance uses `react-native-keyboard-controller` — never RN's built-in `KeyboardAvoidingView`** — `app.json` has `edgeToEdgeEnabled: true` (Expo SDK 54 default). In edge-to-edge mode the Android window no longer resizes when the keyboard opens, so RN's `KeyboardAvoidingView` (`behavior="height"`) has nothing to shrink and the lower fields of a long form stay trapped under the keyboard (the bug shows up on real devices / release APKs, not always in dev). The fix is the Expo-recommended `react-native-keyboard-controller`: a single `<KeyboardProvider>` wraps the app at [app/_layout.tsx](../SubsTrack/app/_layout.tsx) (inside `GestureHandlerRootView`). Two layout shapes, two tools:
    - **Full-scroll forms** — every `*FormSheet` (Customer, Payment, Plan, Branch, Currency, Product, Sale, User), `SaleDetailSheet`, and `LoginScreen` — use `<KeyboardAwareScrollView … bottomOffset={24}>` (drop-in for the old `KeyboardAvoidingView` + `ScrollView` pair). It auto-scrolls the focused `TextInput` above the keyboard and adds bottom inset so the submit button is always reachable.
    - **Centered dialogs / sticky-footer layouts** — `Dropdown`/`AsyncEntityPicker` (centered transparent modal with an internal `FlatList`) and the two signup screens (inner `ScrollView` + a sibling footer button-bar) — use the library's drop-in `<KeyboardAvoidingView behavior="padding">`. `KeyboardAwareScrollView` is wrong here: it would swallow the internal FlatList / detach the sibling footer.
    - Works inside RN `<Modal presentationStyle="pageSheet">` out of the box (keyboard-controller ≥ 1.13 added Android Modal support; iOS always worked) — no nested provider needed. It's a native module: `react-native-reanimated` is the peer dep (already installed), no app.json config plugin is required (autolinked), but adding it means a **dev/EAS rebuild** (not hot-reloadable, not Expo Go — see "Running the Apps").  NativeWind `className` still works on these components via the `jsxImportSource: 'nativewind'` transform.
    - **iOS caveat (verify-on-device):** `KeyboardAwareScrollView` inside a `.pageSheet` UIKit sheet has a documented flicker history ([issue #726](https://github.com/kirillzyusko/react-native-keyboard-controller/issues/726)) and all the form sheets use `<Modal presentationStyle="pageSheet">`. Android (the primary target) is unaffected. If caret-scroll flicker shows up on iOS, the fallback is to use this library's `<KeyboardAvoidingView behavior="padding">` (or `mode="insets"`) for the pageSheet forms instead — smoke-test before shipping iOS.

40. **Edge-function errors must be parsed, not passed straight to `handleError`** — `supabase.functions.invoke` collapses ANY non-2xx response into a generic `FunctionsHttpError` whose `.message` is the useless `"Edge Function returned a non-2xx status code"`. The function's real message lives in the JSON body on `error.context.response` (a `Response`). Repos that invoke edge functions (`UserRepository.create`/`delete`/`updatePassword`) call **`await this.handleFunctionsError(error)`** ([BaseRepository.ts](../SubsTrack/src/core/utils/BaseRepository.ts)) instead of `handleError` — it reads the body via `readFunctionsErrorBody` ([functionsError.ts](../SubsTrack/src/core/utils/functionsError.ts)) and surfaces the server's message (e.g. `"Forbidden: admins can only change passwords of staff users"`), falling back to the generic handling when there's no body. `SignupRepository` uses the same `readFunctionsErrorBody` helper directly (it also needs the `code` field). Any new edge-function call site must follow this pattern, or users see the meaningless "non-2xx" text.

41. **A customer's plans live in `customer_plans` (service lines), not `customers.plan_id`** — `customers.plan_id` was removed. A `CustomerPlan` row = one plan a customer is subscribed to, with its own `start_date` / `cancelled_at`; `plan_id` may be NULL (custom/occasional line). **Plans are managed inline in the customer form** (`CustomerFormSheet` → "Plans" section, create AND edit) — add / change / remove rows there, NOT from the month-grid panel (its line selector is view-only). On save the form creates/updates the customer then calls the `customerPlans` slice's single action `syncLines(customerId, lines, removedIds, tenantId)` ([customerPlanSlice.ts](../SubsTrack/src/state/slices/customer-plans/customerPlanSlice.ts)), which deletes removed lines then creates/updates the rest and calls `get().customers.fetchCustomer(customerId)` so the customer object (single source of truth for its lines) and the grids rebuilt from it refresh. **Remove** = `CustomerPlanService.deleteLine` → hard-delete a line with no payments, else soft-cancel (`active = false`) so payment history is never lost (rule #7). The form keeps **≥1 row** (a plan-less row records custom amounts), so every customer has ≥1 line. Lists/details load lines via `.select('*, customer_plans(*, plans(*))')` → `customer.customerPlans`. `payments.customer_plan_id` is the structural link (which line a payment settles); `payments.plan_id` stays as the price/plan **snapshot** (gotcha #2). Branch is inherited from the customer (no `branch_id` on `customer_plans`; RLS + `BRANCH_SCOPES.customer_plans = inherited`).

42. **A sale's collector cash is `amount_paid`, not `total_amount`** — sales revenue reports sum `total_amount` (what was sold), but the cash a collector physically holds from a sale is `amount_paid` (what was collected; a partial sale leaves the rest as a debt). The **Collector Wallet** therefore reads `amount_paid` for sales (like payments/debt_payments), never `total_amount`. Using `total_amount` would overstate a collector's cash by the unpaid portion. (`WalletService.collectItems` → `saleService.getUnremittedForWallet`.)

43. **Re-paying a voided month must reset `remitted_at`/`remitted_by`** — payments upsert on `(customer_plan_id, billing_month)`, so re-recording a voided month **reuses the same row**. If that row was previously handed over (remitted), the new cash would wrongly look already-remitted. Both write paths null the remittance on re-pay: the web `PaymentRepository.create`/`createMany` upsert object and the offline `buildRow` set `remitted_at: null, remitted_by: null` (and the offline `ON CONFLICT DO UPDATE` copies them via `excluded.*`). This is the reason the Collector Wallet uses **columns on the row** instead of a separate settle/link table — a link keyed by the reused payment id would go stale on re-pay; a column resets cleanly. `custom_debts` has **no** remittance columns (a custom debt isn't collected cash). Editing a payment's `amount_paid` does **not** touch `remitted_at`, so editing an already-remitted payment changes the historical handover figure but not the current wallet (which only counts unremitted rows).

44. **Browser Back closes an open modal on web (mirrors Android hardware-back)** — RN's `Modal` routes Android hardware-back to `onRequestClose`, but RN-**Web**'s `Modal` ignores browser history entirely, so on web the Back button falls through to Expo Router and pops the *route* while the sheet stays open. Fix: `useWebBackDismiss(active, onDismiss)` ([useWebBackDismiss.ts](../SubsTrack/src/shared/hooks/useWebBackDismiss.ts)) — web-only (no-op on native). It keeps a shared LIFO stack of open modals and maintains **one throwaway `history` "sentinel" entry per open modal** (`sentinelCount` tracks `dismissStack.length`). A user Back press pops the topmost sentinel (URL/route unchanged), fires `popstate`, and we dismiss the **topmost** modal; because history holds as many sentinels as there are open modals, the next Back already has its own sentinel waiting — nothing is re-armed in the handler. Closing any other way (Cancel/Save/unmount) drops a sentinel via our own `history.back()` (ignored via a `selfPops` counter). **Scope — only "page-like" / stacking modals use the hook:** the full-page form `<SheetModal>`s (build every page sheet on [SheetModal.tsx](../SubsTrack/src/shared/components/SheetModal.tsx) — it wires this + the standard `slide`/`pageSheet` chrome + native `onRequestClose`; don't hand-roll `<Modal presentationStyle="pageSheet">`) plus the dialogs that can stack on top of them (`ConfirmDialog`, `UpgradePromptModal`) — a dialog **must** trap Back too, else Back would close the sheet *underneath* it and leave the dialog floating. `ConfirmDialog` guards `loading` so Back can't dismiss mid-confirm. **Transient tap-outside popups do NOT use the hook** — the pickers (`Dropdown`, `DatePickerInput`, `CurrencyInput`, `AsyncEntityPicker`) and `ActionMenu` close by tapping their backdrop (and native hardware-back via their own `onRequestClose`). Keeping them out of the history stack is deliberate: they open/close constantly and were the churniest participants, so removing them shrinks the race surface. On web, browser Back while such a popup is the only thing open navigates the route (the popup is a click-outside affordance, not a navigation step); while one sits inside a form sheet, Back closes the sheet and the popup unmounts with it. **Why one sentinel PER modal (not a single re-armed one):** with a single sentinel, each Back consumed it and a fresh one had to be `pushState`d from inside the popstate handler; a **second Back pressed faster than that async re-arm round-trip** navigated against a history with no sentinel left and fell straight through to the route — closing the tab or jumping pages (the reported intermittent "press Back twice on Receive-all and the site closes"). Keeping the sentinel count equal to the modal count means every Back, however fast, always has a sentinel to consume. **Why still reconcile once per tick (not push/pop eagerly):** a modal handoff — one modal closing while another opens/closes in the **same tick** (menu → delete-confirm; confirm → close-parent-sheet) — would otherwise fire an async `history.back()` that overlapped a sibling's `pushState`/`back()`; the browser then popped one step too many and **exited the route**. Coalescing the stack ⇄ history diff into one reconcile per tick (via `queueMicrotask`) means such a handoff nets to "same depth" and touches history not at all, so `back()` and `pushState` never interleave.
